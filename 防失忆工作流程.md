
**重要**：AI恢复上下文后，必须明确说出以下三点才算恢复成功：
- 已完成 5/11 Phase，当前在 Phase 6
- 代码在 `esp32-firmware/`、`smartbrew-server/` 和 `smartbrew-web/` 三个目录
- 下一步是写告警规则引擎 + 告警记录接口 + 告警中心页面

当天开发结束、准备关掉 AI 前，发这段让 AI 写日志：

```
今天的开发结束了，请更新项目文档：
1. PROJECT_STATUS.md — 在"每日日志"末尾追加今日日志（日期、完成事项、Git提交、遇到的问题、明日计划）
2. PROJECT_ROADMAP.md — 已完成子任务的复选框标记为 [x]，完成的 Phase 改为 ✅ 并更新进度总览
3. PROJECT_STATUS.md — 顶部"最后更新"时间改为今天
4. PROJECT_ROADMAP.md — "下次恢复"那一行更新为当前阶段
5. Git 提交所有变更（代码 + 文档分开提交）
```

## 十一、关键架构决策（AI 必须知道）

| 决策点 | 决定 | 说明 |
|--------|------|------|
| ESP32框架 | Arduino (PlatformIO) | 非 ESP-IDF，串口波特率 115200 |
| MQTT客户端 | Eclipse Paho (Java) / PubSubClient (C++) | 服务端和固件端不同库 |
| MQTT Topic | `smartbrew/device/{deviceId}/data` 等 | 订阅用 `smartbrew/+/data` 通配符 |
| ORM | MyBatis Plus 3.5.6 | 非 JPA，BaseMapper + LambdaQueryWrapper |
| 连接池 | Druid | application.yml 中配置 |
| Redis序列化 | GenericJackson2Json | Key 用 String，Value 用 JSON |
| 在线状态 | Redis TTL=120s + DB 定时同步 | 双重保障，心跳刷新 TTL |
| API响应 | `ApiResult<T>` 统一格式 | `{code: 200, message: "success", data: T}` |
| 自动建表 | `spring.sql.init.mode=always` | schema.sql 含 CREATE TABLE IF NOT EXISTS |
| 设备注册 | 支持手动注册 + 心跳自动注册 | `MqttSubscriberService.handleHeartbeat()` 中实现 |
| 心跳超时 | @Scheduled 每60s扫描，120s无心跳→离线 | `HeartbeatTimeoutTask` |
| 温度阈值 | >25℃开风扇，<18℃开加热 | ESP32端已预留 GPIO，服务端 Phase 8 实现规则引擎 |
